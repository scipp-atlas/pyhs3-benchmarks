# Benchmark Results

Every benchmark execution produces structured outputs that can be used for analysis, visualization, validation, and long-term performance tracking.

All benchmark outputs are deterministic and organized using a consistent directory structure, allowing large benchmark campaigns to be reproduced and compared across repository revisions.

---

# Output Directories

Benchmark outputs are organized into two primary directories.

```text
results/
plots/
```

The `results` directory stores structured benchmark reports, while `plots` contains generated visualizations.

Keeping numerical results separate from figures allows plots to be regenerated without rerunning benchmarks.

---

# Results Directory

The matrix runner stores benchmark outputs using the following hierarchy.

```text
results/

    <benchmark>/

        <workspace>/

            repeat_000/

                <benchmark>_result.json

            repeat_001/

                <benchmark>_result.json

            ...

        ...

    matrix_summary.json
```

This structure scales naturally from a single benchmark execution to large benchmark campaigns involving dozens of workspaces.

Each benchmark execution is completely independent from every other execution.

---

# Benchmark Reports

Each benchmark generates a JSON report describing the execution.

Although the exact contents depend on the benchmark, reports typically contain

- benchmark metadata;
- benchmark configuration;
- workspace information;
- execution status;
- timing statistics;
- memory statistics;
- validation results;
- benchmark-specific measurements.

These JSON files serve as the primary machine-readable output of the repository.

---

# Matrix Summary

After completing a benchmark campaign, the matrix runner generates

```text
matrix_summary.json
```

This file summarizes the entire execution.

Typical information includes

- executed benchmarks;
- processed workspaces;
- execution status;
- failed runs;
- benchmark locations.

The summary provides a convenient entry point for automated analysis and reporting.

---

# Generated Figures

Plots generated by the benchmark framework are written to

```text
plots/
```

using a directory hierarchy mirroring the corresponding benchmark results.

Typical figures include

- execution time comparisons;
- memory usage;
- scaling behaviour;
- framework comparisons;
- likelihood scans;
- benchmark overview plots.

Since plots are generated from benchmark reports, they can be recreated at any time without repeating benchmark execution.

---

# Benchmark Overview

The repository includes a dedicated overview generator that summarizes complete benchmark campaigns.

Rather than visualizing a single benchmark, the overview aggregates results across multiple benchmark suites and presents them using consistent reporting conventions.

Typical overview plots include

- performance summaries;
- timing comparisons;
- memory summaries.

These reports provide a high-level view of benchmark performance across the entire repository.

---

# Repeated Executions

Many benchmarks execute multiple repeated measurements.

Each repetition is stored independently.

```text
repeat_000/

repeat_001/

repeat_002/
```

Storing repetitions separately preserves the complete benchmark history and simplifies debugging when individual executions fail.

---

# Benchmark Status

Every benchmark execution records its completion status.

Typical values include

- success
- failed
- timeout
- error
- skipped

Status information allows automated tools to distinguish successful benchmark results from incomplete executions.

---

# Reproducibility

Benchmark outputs are intended to be reproducible.

Running the same benchmark configuration on identical benchmark inputs should produce equivalent numerical results, while execution times may vary slightly depending on the hardware and execution environment.

For this reason, benchmark reports always include the benchmark configuration together with the measured statistics.

---

# Typical Workflow

A typical benchmark workflow looks as follows.

```text
Benchmark
     │
     ▼
JSON Report
     │
     ▼
Matrix Summary
     │
     ▼
Overview Builder
     │
     ▼
Plots
     │
     ▼
Documentation
```

This workflow separates benchmark execution from visualization, making it possible to regenerate figures and reports without repeating expensive benchmark runs.

---

# Using Benchmark Results

Benchmark reports are intended to support several use cases.

- performance regression detection;
- optimization studies;
- framework comparisons;
- scalability analysis;
- publication-quality figures;
- automated reporting.

Because every benchmark follows the same reporting conventions, analysis tools can operate uniformly across the repository without requiring benchmark-specific logic.
